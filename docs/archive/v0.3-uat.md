# BranchNuX v0.3 — UAT Layer

> **Status:** Scaffold released. Full sign-off UI and RTM automation arrive in v0.3 release.

## Taxonomy: BR-XX vs R-XX vs TC-XX vs BLOCKED-*

BranchNuX uses a three-tier requirement hierarchy plus a special status class:

| Symbol | Meaning | Scope | Who writes it |
|--------|---------|-------|---------------|
| `BR-XX` | **Business Requirement** — a business outcome the system must deliver | Product / business | Product owner or lead |
| `R-XX` | **Functional Requirement** — a specific capability that implements a BR | Engineering | Engineering spec |
| `TC-XX` | **Test Case** — a specific Given/When/Then verifiable scenario for an R-ID | QA / engineering | Test author |
| `BLOCKED-*` | A TC or R-ID that cannot be executed/verified due to an external dependency | Any | Whoever finds the block |

One BR can map to many R-IDs. One R-ID can map to many TCs. The RTM makes this graph explicit.

```
BR-01: Stakeholder can self-serve account recovery
  ├─ R-99: WebAuthn fallback (recovery codes)
  │  ├─ LOGIN-15 (recovery code accepted)
  │  └─ LOGIN-16 (recovery code one-time use)
  └─ R-100: Admin-mediated reset
     └─ ADM-SET-05 (admin can reset on request)
```

## BR-XX Uniqueness Lint

BR-IDs must be unique across the project. The `branchnux validate` command enforces this:

```bash
branchnux validate requirements/ --check br-uniqueness
```

The lint rule rejects:
- Duplicate `## BR-XX` headings in `BUSINESS_REQUIREMENTS.md`
- BR-IDs that shadow existing R-IDs (e.g. `BR-01` when `R-01` exists — different namespaces, but confusing)

## Sign-Off Workflow

### Step 1: Scaffold a BR

```bash
branchnux br init BR-01
```

Creates or appends a `## BR-01` section in `requirements/BUSINESS_REQUIREMENTS.md` with:
- Frontmatter: `uat_status`, `owner`, `stakeholders`, `approval_required`
- Sections: Business outcome / Acceptance criteria / Linked R-IDs / Linked TC-IDs / Stakeholder sign-off matrix

### Step 2: Link to functional requirements

```bash
branchnux br link BR-01 R-99,R-100
```

Updates the `### Linked R-IDs` section in `BUSINESS_REQUIREMENTS.md`.

### Step 3: Generate the UAT traceability matrix

```bash
branchnux br rtm
```

Writes `requirements/UAT_TRACEABILITY.md` — a markdown table mapping BR-XX → R-XX → TC-XX.

### Step 4: Run tests, then sign off

```bash
# Run the test pass (with environment suffix)
branchnux run login --env staging

# Interactive sign-off
branchnux sign 2026-04-26_login-staging
```

The interactive prompt collects:
- **Reviewer name**
- **Reviewer role** (e.g. "QA Lead", "Product Owner", "Auditor")
- **TC-ID** (which test case you are signing off)
- **Status**: `accepted` | `rejected` | `needs-rework`
- **Justification** (required for `rejected` and `needs-rework`)

### Step 5: Batch reject a TC

```bash
branchnux sign 2026-04-26_login-staging --reject LOGIN-15
```

Prompts for name, role, and justification; status is set to `rejected` automatically.

## E-Signature: What It Is and What It Is Not

Each sign-off entry is signed with HMAC-SHA256 computed over:

```
tc_id + "|" + status + "|" + reviewer_id + "|" + timestamp
```

using the project's `UAT_SECRET` environment variable as the key.

**What this provides:**
- Tamper-evident audit trail (chain breaks if any entry is modified or removed)
- Proof of which reviewer signed what, when, with what status
- Suitable for SOC 2 Type II evidence, ISO 27001 A.12.7 audit logging, and internal compliance reviews

**What this does NOT provide:**
- Court-admissible e-signature under eIDAS (EU), ESIGN (US), or UETA (US)
- Non-repudiation in a legal sense (HMAC key is a shared secret, not asymmetric)
- Binding contractual signature

> For legally binding e-signatures, engage legal counsel and a qualified trust service provider (e.g. DocuSign, Adobe Sign, qualified e-signature under eIDAS Article 26).

## Hash-Chained JSONL Log

Sign-off entries are appended to `<surface>/uat-log.jsonl`.

Each entry includes:
```json
{
  "tc_id": "LOGIN-15",
  "status": "accepted",
  "reviewer": "Jane Smith",
  "reviewer_role": "QA Lead",
  "justification": "",
  "ts": "2026-04-26T14:32:00.000Z",
  "prev_hash": "a3f1c2...",
  "signature": "7b9e4d..."
}
```

- `prev_hash` — HMAC-SHA256 of the raw JSON string of the previous entry (or `hmac(secret, '')` for the first entry)
- `signature` — HMAC-SHA256 of `tc_id|status|reviewer|ts`

**Verify the chain:**
```bash
branchnux sign --verify 2026-04-26_login-staging
```

Reports `valid: true` or identifies the exact line where the chain breaks.

## Stale Acceptance Detection

Sign-offs older than 90 days are considered stale and require re-attestation.

```bash
branchnux validate requirements/ --check stale-uat
```

Output example:
```
WARN: BR-01 / LOGIN-15 — accepted by Jane Smith on 2025-11-01 (177 days ago — re-attest required)
```

The lint rule is configurable in `branchnux.config.mjs`:
```js
export default {
  uat: {
    staleAfterDays: 90,  // default
  }
}
```

## Full BR-XX → R-XX → TC-XX Example Walkthrough

**Scenario:** A project requires that users can recover their account if they lose their MFA device.

### 1. Create the BR

```bash
branchnux br init BR-01
# Edit requirements/BUSINESS_REQUIREMENTS.md:
# Business Outcome: "A user who has lost their MFA device can regain account access
#   within 24 hours without requiring manual intervention from an administrator,
#   provided they can verify their identity via backup codes."
```

### 2. Write the functional requirements

In `requirements/REQUIREMENTS.md`:
```markdown
## R-99 — WebAuthn fallback (recovery codes)
**Status:** DONE

## R-100 — Admin-mediated MFA reset
**Status:** DONE
```

### 3. Link BR to R-IDs

```bash
branchnux br link BR-01 R-99,R-100
```

### 4. Scaffold the test pass

```bash
branchnux run login --env staging
# Folder created: testing-log/2026-04-26_login-staging/
```

### 5. Write and run TCs

In `test-plan.md`:
```
| LOGIN-15 | Recovery code accepted on first use | PASS | ... |
| LOGIN-16 | Recovery code rejected on second use (one-time) | PASS | ... |
```

### 6. Sign off

```bash
branchnux sign 2026-04-26_login-staging
# Reviewer name: Jane Smith
# Reviewer role: QA Lead
# TC-ID: LOGIN-15
# Status: accepted
# Justification: (blank — accepted)
```

### 7. Generate the RTM

```bash
branchnux br rtm
# Writes requirements/UAT_TRACEABILITY.md:
# | BR-01 | Account MFA recovery | accepted | R-99, R-100 | LOGIN-15, LOGIN-16 |
```

The RTM is the single source of truth for "what business requirement does each test cover."
